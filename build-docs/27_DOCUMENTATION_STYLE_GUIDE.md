## Voice and Tone

- **Professional but accessible** — Avoid jargon without definition
- **Direct and concise** — Use active voice, imperative for instructions
- **Consistent terminology** — Follow [25_CANONICAL_TERMINOLOGY.md](25_CANONICAL_TERMINOLOGY.md)

## Structure

Every document must include:
1. Version header table
2. Purpose section
3. Scope (covers/does not cover)
4. Audience
5. Body content with headings
6. Engineering Notes
7. References
8. Cross References
9. Best Practices
10. Common Mistakes
11. Review Checklist

## Formatting

- Use ATX-style headings (# not =)
- Tables for structured data
- Code blocks with language tags
- Mermaid diagrams for architecture
- Internal links in [description](file.md) format
