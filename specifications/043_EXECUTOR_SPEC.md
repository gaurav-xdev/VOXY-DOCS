# 043 - Executor Specification

## Purpose

Define the Executor subsystem that executes plans generated by the Planner, managing the lifecycle of each action, handling results, and coordinating with downstream subsystems.

## Scope

- Plan step execution
- Action dispatch to appropriate subsystems
- Result collection and aggregation
- Error handling and escalation
- Parallel execution of independent steps
- Execution telemetry

## Responsibilities

- Execute plan steps in dependency order
- Dispatch actions to correct subsystems
- Collect and aggregate results
- Handle step failures
- Support parallel execution where safe
- Report execution progress

## Non-Goals

- Plan generation (Planner responsibility)
- Tool implementation (individual subsystems)
- UI rendering (UI layer)

## Architecture Position

Executor sits between Planner and all action-capable subsystems (Tool Calling, UI Automation, etc.).

## Inputs

- Validated plans from Planner
- Step execution results from subsystems
- Timeout and cancellation signals

## Outputs

- Step completion results
- Execution progress events
- Final plan results
- Error reports

## Interfaces

```rust
pub struct Executor {
    pub fn execute(plan: &Plan) -> Result<ExecutionResult, ExecutionError>;
    pub fn execute_step(step: &PlanStep) -> Result<StepResult, ExecutionError>;
    pub fn cancel_execution(plan_id: UUID);
    pub fn get_progress(plan_id: UUID) -> ExecutionProgress;
}
```

## Data Models

### ExecutionResult

| Field | Type | Description |
|-------|------|-------------|
| plan_id | UUID | Plan identifier |
| status | ExecutionStatus | Success, Partial, Failed |
| step_results | Vec<StepResult> | Per-step results |
| duration_ms | u64 | Total execution time |
| summary | String | Human-readable summary |

### ExecutionStatus

| Variant | Description |
|---------|-------------|
| `Success` | All steps completed |
| `Partial` | Some steps failed, plan continued |
| `Failed` | Critical step failed |
| `Cancelled` | User cancelled |
| `Timeout` | Plan exceeded timeout |

### StepResult

| Field | Type | Description |
|-------|------|-------------|
| step_id | UUID | Step identifier |
| status | StepStatus | Success, Failed, Skipped |
| output | Option<String> | Step output |
| error | Option<ExecutionError> | Error if failed |
| duration_ms | u64 | Step execution time |

## State Management

- Execution state is maintained per plan
- Progress is broadcast via Event Bus
- Results are stored in Working Memory

## Threading Model

Independent steps execute in parallel on Tokio runtime. Dependent steps wait for prerequisites.

## IPC Requirements

- No IPC for execution
- Progress may be queried via IPC

## Event Definitions

| Event | Payload | Description |
|-------|---------|-------------|
| `execution.started` | plan_id | Execution started |
| `execution.progress` | ExecutionProgress | Progress update |
| `execution.step.completed` | StepResult | Step finished |
| `execution.completed` | ExecutionResult | Execution finished |
| `execution.cancelled` | plan_id | Execution cancelled |

## Error Handling

| Code | Description | Recovery |
|------|-------------|----------|
| `EX001` | Step dispatch failed | Retry, then fail step |
| `EX002` | Step timeout | Cancel step, retry |
| `EX003` | Subsystem unavailable | Queue, retry later |
| `EX004` | Permission denied | Skip step, notify user |
| `EX005` | Execution deadlock | Detect, break, alert |

## Retry Strategy

- Step dispatch: 2 retries
- Subsystem unavailable: Retry with backoff
- Timeout: Cancel and retry once

## Recovery Strategy

- Step failure: Continue if non-critical, replan if critical
- Subsystem failure: Queue for later
- Deadlock: Break and alert

## Security Requirements

- Actions validated against capability tokens
- Sensitive actions require confirmation
- Execution sandboxed where possible

## Performance Targets

| Metric | Target |
|--------|--------|
| Step dispatch latency | < 10ms |
| Parallel step overhead | < 5ms |
| Progress update latency | < 100ms |

## Resource Limits

| Resource | Limit |
|----------|-------|
| Concurrent plans | 3 |
| Concurrent steps | 5 |
| Step timeout | 300s |
